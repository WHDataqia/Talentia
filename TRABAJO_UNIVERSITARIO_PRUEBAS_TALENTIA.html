1. Introducción

El aseguramiento de la calidad del software es una disciplina fundamental en la ingeniería informática. Garantizar que un sistema funcione según lo especificado requiere aplicar metodologías de pruebas rigurosas que cubran tanto la lógica interna del código como el comportamiento visible por el usuario. Las dos técnicas más representativas de esta disciplina son las pruebas de caja blanca, orientadas a la estructura interna del código, y las pruebas de caja negra, orientadas al comportamiento externo del sistema sin conocimiento de su implementación.

El presente trabajo aplica ambas técnicas sobre Talentia, un sistema web de gestión del desempeño desarrollado de forma autónoma para digitalizar el proceso de evaluación por competencias de una organización del sector hotelero. Antes de la implementación del sistema, el proceso se realizaba mediante documentos Word que se distribuían por correo, generando dispersión de la información, inconsistencias y ausencia de trazabilidad histórica. Talentia resuelve estas problemáticas ofreciendo un flujo centralizado, roles diferenciados y persistencia de datos en PostgreSQL.

El documento está estructurado en cuatro secciones principales: una introducción al contexto y motivación, la definición de los objetivos de las pruebas, el desarrollo técnico de las pruebas de caja blanca y caja negra junto con el modelo de negocio organizacional, y finalmente las conclusiones extraídas del ejercicio.

2. Objetivos

2.1 Objetivo General

Aplicar las metodologías de prueba de caja blanca y caja negra sobre el software Talentia para verificar la correcta implementación de su lógica de negocio y validar el comportamiento esperado desde la perspectiva del usuario final; complementando el análisis con un modelo de negocio que contextualiza el referente organizacional donde opera el sistema.

2.2 Objetivos Específicos

Objetivo 1

Identificar y documentar las rutas de prueba de la lógica de autenticación y evaluación del backend mediante la técnica de caja blanca (cobertura de caminos y condiciones).

Objetivo 2

Diseñar casos de prueba de caja negra para las interfaces de usuario críticas del sistema (login, evaluación de competencias, historial), verificando entradas válidas e inválidas.

Objetivo 3

Evaluar la cobertura de las pruebas aplicadas e identificar las áreas de mejora en la calidad del software.

Objetivo 4

Construir un modelo de negocio (Canvas) que represente la propuesta de valor, segmentos de clientes y estructura financiera del sistema Talentia en su contexto organizacional.

3. Desarrollo

3.1 Descripción del Software Talentia

Talentia es una aplicación web full-stack desarrollada para gestionar el ciclo completo de evaluación de desempeño por competencias. Su arquitectura se divide en tres capas claramente diferenciadas:

  • Frontend: Interfaz construida en HTML5, CSS3 y JavaScript vanilla. Incluye las pantallas de inicio de sesión (login.html), autoevaluación (autoevaluacion.html), creación de evaluaciones (crear-evaluacion.html), historial (historial.html), gestión de empleados (empleados.html) y panel de administración (admin.html).
  • Backend: API REST implementada en Python con el framework Flask. Los módulos principales son app.py (rutas y lógica de negocio central), auth.py (autenticación y autorización), competencias_db.py (acceso a datos de competencias), rutas_competencias.py (endpoints específicos de competencias) y migrations.py (migraciones de base de datos).
  • Base de datos: PostgreSQL con un modelo relacional que incluye las tablas empleados, evaluaciones, competencias, descriptores_competencia, comportamientos y evaluaciones_competencia. Se utiliza un pool de conexiones (5–30 conexiones) para soportar concurrencia.

Competencias evaluadas

El sistema evalúa ocho competencias organizacionales en una escala de 1 a 5:

  1. Gestión Operativa
  2. Innovación y Mejora Continua
  3. Servicio al Cliente
  4. Trabajo en Equipo y Colaboración
  5. Liderazgo e Influencia
  6. Comunicación Efectiva
  7. Orientación a Resultados
  8. Gestión del Cambio y Adaptabilidad

Roles de usuario

  • Administrador: Gestión completa de empleados, evaluaciones, períodos y maestras.
  • Jefe / Evaluador: Evalúa a sus subordinados directos; visualiza su equipo.
  • Empleado: Realiza autoevaluación; consulta su propio historial.

3.2 Pruebas de Caja Blanca (Backend)

Las pruebas de caja blanca —también llamadas pruebas estructurales o de caja de cristal— se basan en el conocimiento de la estructura interna del código. El objetivo es garantizar que todos los caminos de ejecución, condiciones de frontera y ciclos sean ejercitados al menos una vez. El criterio de cobertura aplicado en este análisis es la cobertura de condiciones/decisiones (MC/DC), complementada con análisis del grafo de flujo de control de las funciones críticas.

Referencia de técnica: Las pruebas de caja blanca corresponden a la norma IEEE 829-2008 y se aplican sobre el código fuente del módulo auth.py, la ruta POST /api/login de app.py y la ruta POST /api/evaluaciones/nivel.

3.2.1 Módulo de Autenticación — auth.py

La función autenticar_usuario(conn, email, contrasena) contiene la siguiente lógica de control: 

# Fragmento simplificado de auth.py
def autenticar_usuario(conn, email, contrasena):
    cursor = conn.cursor()
    cursor.execute(
        'SELECT * FROM empleados WHERE correo_corporativo = %s AND activo = true',
        (email,)
    )
    usuario = cursor.fetchone()

    if not usuario:                                   # ← Decisión 1
        return None, 'Usuario o contraseña incorrectos'

    if not verificar_contrasena(contrasena, usuario['contrasena_hash']):  # ← Decisión 2
        return None, 'Usuario o contraseña incorrectos'

    return dict(usuario), None

Grafo de flujo de control — autenticar_usuario

Se identifican los siguientes nodos y decisiones:

  • N1: Inicio – ejecutar consulta SQL.
  • D1: ¿El usuario existe en BD? (True → N3, False → N2)
  • N2: Retornar None, 'Usuario o contraseña incorrectos' (camino error usuario).
  • D2: ¿La contraseña hash coincide? (True → N4, False → N2b)
  • N4: Retornar dict(usuario), None (éxito).

La complejidad ciclomática calculada es V(G) = 3, lo que requiere mínimo 3 caminos de prueba independientes.

3.2.2 Función autenticar_empleado_por_cedula

Esta función implementa un mecanismo de fallback: primero busca por el campo cedula y, si no encuentra registro, reintenta la búsqueda por el campo identificacion. Esto cubre el caso donde la cédula fue modificada erróneamente desde el panel de administración. 

# Fragmento de auth.py
cursor.execute('SELECT * FROM empleados WHERE cedula = %s AND activo = true', (cedula,))
usuario = cursor.fetchone()

# Fallback: buscar por identificacion
if not usuario:                                        # ← Decisión 3
    cursor.execute(
        'SELECT * FROM empleados WHERE identificacion = %s AND activo = true', (cedula,)
    )
    usuario = cursor.fetchone()

if not usuario:                                        # ← Decisión 4
    return None, 'Cédula o contraseña incorrectos'

if not usuario['contrasena_hash']:                     # ← Decisión 5
    return None, 'Usuario no tiene contraseña configurada...'

if not verificar_contrasena(contrasena, usuario['contrasena_hash']):  # ← Decisión 6
    return None, 'Cédula o contraseña incorrectos'

return dict(usuario), None

Complejidad ciclomática: V(G) = 5

3.4 Modelo de Negocio

El modelo de negocio describe de forma estructurada cómo una organización crea, entrega y captura valor. Para este análisis se emplea el Business Model Canvas (Osterwalder & Pigneur, 2010), aplicado al contexto organizacional donde Talentia fue desarrollado e implementado.

Contexto organizacional

Talentia fue desarrollado para una empresa del sector hotelero con múltiples unidades operativas. La organización contaba con un proceso de evaluación de desempeño manual, basado en documentos Word distribuidos por correo, lo que dificultaba la consolidación de resultados, el seguimiento histórico y la toma de decisiones basada en datos sobre el talento humano. La necesidad de digitalizar y centralizar este proceso constituyó la oportunidad de negocio que dio origen al software.

Recursos Clave

Propuesta de Valor

  • Centralización: Una sola plataforma para todas las evaluaciones de desempeño de la organización
  • Eficiencia: Reducción del tiempo del proceso (de días a horas)
  • Trazabilidad: Historial por empleado y período accesible en cualquier momento
  • Objetividad: Evaluación estandarizada por competencias con descriptores conductuales

  • Escalabilidad interna: El sistema puede replicarse a otras unidades de negocio del grupo hotelero sin costo adicional

4. Conclusiones

🔬

Complementariedad de técnicas

Las pruebas de caja blanca y caja negra son técnicas complementarias, no excluyentes. La caja blanca garantizó que todos los caminos de decisión en los módulos de autenticación y evaluación fueran ejercitados, mientras que la caja negra verificó que el comportamiento observable desde el navegador coincidiera con las especificaciones de usuario.

🛡️

Calidad y seguridad demostradas

Los 44 casos de prueba documentados pasaron satisfactoriamente en la versión actual del sistema. Los controles de acceso por rol, el uso de contraseñas hasheadas con PBKDF2-SHA256, las consultas parametrizadas contra SQL injection y la validación de entradas en frontend ofrecen un nivel de seguridad adecuado para el contexto de uso.

📈

Valor de negocio medible

El modelo de negocio Canvas evidencia que Talentia genera un valor cuantificable para la organización: reducción superior al 70% en el tiempo dedicado al proceso de evaluación, eliminación de errores de transcripción y habilitación de toma de decisiones basada en datos históricos por competencia y período.

🔄

Deuda técnica identificada

El análisis de pruebas reveló oportunidades de mejora: implementar pruebas de mutación para detectar código "zombie", agregar pruebas automatizadas de regresión con pytest, y documentar formalmente la API REST con OpenAPI/Swagger para facilitar el mantenimiento por otros desarrolladores.

🏗️

SDLC como marco integrador

La aplicación del ciclo de vida del desarrollo de software (SDLC) —planificación, diseño, implementación, pruebas, despliegue y mantenimiento— demostró ser la base para construir un sistema estable y sostenible. Las pruebas no son una fase aislada sino un proceso continuo que atraviesa todo el ciclo.

🌐

Escalabilidad del modelo

El modelo de negocio Canvas revela que la propuesta de valor de Talentia es escalable más allá de una única sede u organización. Su arquitectura técnica basada en stack open source, su diseño modular y la ausencia de dependencias de licencias propietarias facilitan su replicación o evolución hacia un modelo SaaS para el sector hospitalidad.
