1. Introducción a los Métodos de Prueba

En el ciclo de vida de Talentia se aplican dos metodologías complementarias de verificación. Las Pruebas de Caja Negra se ejecutan sobre la capa de presentación (HTML/JS), evaluando el comportamiento visible sin acceder al código fuente. Las Pruebas de Caja Blanca inspeccionan el código Python/Flask del backend, garantizando cobertura de cada rama lógica.

■

Caja Negra — Frontend

  • Se prueba el comportamiento externo de cada pantalla HTML
  • El evaluador interactúa con formularios, botones y mensajes de error
  • No se requiere conocer el código JavaScript interno
  • Enfoque: entradas → salidas esperadas
  • Cubre: validaciones de formulario, navegación, mensajes de estado
  • Técnicas: partición de equivalencia, valores límite, tablas de decisión
□

Caja Blanca — Backend

  • Se prueba la lógica interna de las funciones Python/Flask
  • El evaluador conoce el código fuente y diseña pruebas por rama
  • Garantiza cobertura de cada condicional (if/else/except)
  • Enfoque: caminos lógicos → resultado correcto por ruta
  • Cubre: autenticación, middleware JWT, pool de BD
  • Técnicas: cobertura de sentencias, ramas, caminos básicos

Comparación de métodos aplicados en Talentia

Dimensión ■ Caja Negra  Frontend □ Caja Blanca  Backend
Capa evaluada HTML, CSS, JavaScript — pantallas del navegador Python, Flask, psycopg2 — lógica del servidor
Entradas Acciones de usuario: clics, escritura en campos, envío de formularios Argumentos de función, estado de BD, variables de entorno
Salidas esperadas Mensajes de error UI, redirecciones, cambios de estado visual Tuplas de retorno (datos, None) o (None, error), respuestas HTTP
Conocimiento del código No requerido (perspectiva del usuario final) Obligatorio (se diseña por ramas del flujo de control)
Herramientas Navegador, DevTools, casos manuales / Selenium pytest, unittest.mock, cobertura con coverage.py

3. Pruebas de Caja Blanca — Backend (Python / Flask)

Las pruebas de caja blanca analizan el grafo de flujo de control de las funciones críticas del backend. Para cada función se identifican los caminos básicos independientes (método de McCabe) y se diseña un caso de prueba por camino, asegurando cobertura total de ramas.

Herramientas de prueba utilizadas pytest para ejecución • unittest.mock.patch para aislar la BD • coverage.py --branch para medir cobertura de ramas del código Python.

3.1 Función autenticar_usuario(conn, email, contrasena)

backend/auth.py

Autentica administradores y evaluadores mediante email. Retorna (usuario_dict, None) en éxito o (None, mensaje_error) en fallo. La tabla de decisiones protege contra enumeración: error genérico tanto si el usuario no existe como si la contraseña es incorrecta.

def autenticar_usuario(conn, email, contrasena): try: cur = conn.cursor(cursor_factory=RealDictCursor) cur.execute("SELECT * FROM usuarios WHERE email=%s", (email,)) usuario = cur.fetchone() if not usuario: # Rama 1: usuario no encontrado return None, "Usuario o contraseña incorrectos" stored = usuario["contrasena_hash"] if not verificar_contrasena(contrasena, stored): # Rama 2: hash incorrecto return None, "Usuario o contraseña incorrectos" return dict(usuario), None # Rama 3: éxito except Exception as e: # Rama 4: excepción BD return None, f"Error en autenticación: {e}"

Complejidad ciclomática V(G) = 4 (cuatro caminos independientes)

Camino 1

Email no existe en BD → return (None, "incorrectos")

Camino 2

Email existe — hash no coincide → return (None, "incorrectos")

Camino 3

Email existe — hash correcto → return (dict_usuario, None)

Camino 4

Excepción en BD → return (None, "Error en autenticación: ...")

3.3 Decorador jwt_required_if_hardening_enabled()

backend/app.py

Aplica la validación JWT solo si la variable de entorno SECURITY_HARDENING está activa. Permite despliegues en desarrollo sin token mientras protege producción. V(G) = 2.
